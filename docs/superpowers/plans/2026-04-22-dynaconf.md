# Dynaconf Integration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace hardcoded `SECRET_KEY`, `DEBUG`, and `ALLOWED_HOSTS` in `settings.py` with layered dynaconf config: shared YAML defaults, per-environment YAML overrides, and `DJANGO_*` env var overrides.

**Architecture:** `DjangoDynaconf` patches Django's settings module at import time. `ENV_FOR_DYNACONF` (set per Docker Compose file) selects the per-env YAML file (`settings.{env}.yaml`) loaded on top of shared defaults (`settings.yaml`). Env vars prefixed `DJANGO_` override everything. `DATABASES` stays in `settings.py` as Python to avoid dynaconf's nested-key uppercase limitation.

**Tech Stack:** dynaconf, YAML, Docker Compose `environment:` blocks

- [ ] **Step 2: Update `docker-compose.yml` — add environment block**

  Add after the `build:` block (before `volumes:`):

  ```yaml
  services:
    app:
      build:
        context: .
        dockerfile: Dockerfile
        target: dev
        args:
          APP_UID: 1000
          APP_GID: 1001
      environment:
        - ENV_FOR_DYNACONF=development
      volumes:
        - ./project:/opt/project/project
        - ./src:/opt/project/src
        - ./db.sqlite3:/opt/project/db.sqlite3
  ```

- [ ] **Step 4: Update `docker-compose.prod.yml` — add environment block**

  ```yaml

      build:
        context: .
        dockerfile: Dockerfile

      restart: unless-stopped

        - ENV_FOR_DYNACONF=production
  ```
